Product requirements for the legal retrieval and verification workflow.

This section is sourced from the PRD in System Design + PRD.txt. It focuses on target users, product goals, functional requirements, non-functional requirements, user flows, and MVP boundaries.

Product overview

From PRD

VerifRAG is a legal research assistant designed to reduce hallucinations in LLM outputs by verifying generated claims against trusted legal documents. The product is aimed primarily at paralegals, who routinely work through large volumes of cases, filings, exhibits, and supporting materials while preparing reliable summaries for attorneys and legal teams.

Primary goals

From PRD
  • Reduce hallucinated legal responses.
  • Provide citation-backed answers.
  • Surface contradictions clearly and quickly.
  • Preserve confidentiality through a local-first workflow.

User personas and MVP capabilities

From PRD + tech spec
Persona / capability Description Why it matters
Paralegal Uploads legal documents, submits research questions, reviews verified responses, and turns large document sets into usable legal summaries. Best-fit primary user because paralegals handle high volumes of relevant documents and need fast, citation-backed summaries they can trust.
System administrator Manages user accounts and maintains the local legal corpus. Keeps the system usable in a firm environment.
Feature A Upload, manage, and encode legal documents into the vector store. Creates the searchable legal corpus.
Feature B-C Submit natural-language queries and generate answers from retrieved context. Core research workflow.
Feature D-E Decompose answers into atomic claims and verify each claim with NLI. Differentiates the product from standard RAG.
Feature F Display confidence indicators and claim-level warnings in the UI. Makes verification usable under time pressure.

Paralegals are the strongest audience for the MVP because they sit closest to the document-heavy workflow: organizing evidence, reviewing source material, and producing concise research summaries under time pressure. A claim-level verification system directly supports that work by reducing the risk of inaccurate summaries making it upstream.

Functional requirements

From PRD
  • Users and access controls with session isolation.
  • PDF and DOCX ingestion, validation, parsing, chunking, and indexing.
  • Natural-language query submission with retrieval filters.
  • Response generation with citations and stored history.
  • Claim decomposition, NLI verification, and claim-level warnings.
  • Upload, query, results, evidence drill-down, logging, timeout, and audit support.

Non-functional requirements and scope

From PRD
  • NLI F1 target of at least 0.80.
  • All displayed citations must map to valid local chunks.
  • Response plus verification target of 90 to 120 seconds.
  • Traffic-light UI and visible processing indicators.
  • Out of scope for MVP: multi-user collaborative sessions, cloud-hosted deployment, automated legal advice, and undefined external integrations.

Architecture principles

From system design
  • Local processing and security-first design.
  • Deterministic retrieval with authority-aware ranking.
  • Strict claim-level verification after generation, not during generation.
  • Loosely coupled modules that are independently testable.

Error handling and resilience

From technical spec
  • Monitor GPU and VRAM exhaustion and pause verification rather than crashing.
  • Flag parsing failures with retry or skip behavior for unreadable legal documents.
  • Expose stop-generation controls for long-running model calls.
  • Mark ambiguous verification as insufficient evidence instead of guessing.
Detailed Design

The deep dive: algorithms, data model, and service contracts.

This section answers the assignment requirement for component-level logic, database structure, and API details. Where the summary names interfaces but does not define the exact payload, the site labels those examples as draft contracts.

Five verification algorithms

From summary

1. Evidence aggregation

Combines entailment scores across passages with authority weighting so high-value precedent can dominate noisy context.

2. Claim decomposition

Extracts citation, statutory, holding, and factual claims using regex, spaCy, and optional LLM fallback.

3. Contradiction detection

Clusters similar chunks before pairwise NLI so conflict checks stay efficient and can surface circuit splits.

4. Verdict classification

Maps scores to verified, supported, weak, unsupported, or contradicted with asymmetric legal-domain thresholds.

5. Citation verification

Parses citations, confirms authority existence, extracts holdings, checks accuracy, and flags bad law risk.

Entity relationship diagram

From ERD

The ERD shows a conversation-centered relational model: users own conversations, conversations contain messages and interactions, and each interaction can emit verified claims, contradiction records, and source citations.

Relational schema in the ERD

From ERD
Entity Key fields shown Purpose
users id, email, password_hash, created_at Authentication and ownership root for the rest of the relational data.
conversations id, user_id, title, created_at, updated_at Stores the top-level research thread a user is working in.
messages id, conversation_id, content, created_at Preserves the visible chat history inside a conversation.
interactions id, conversation_id, user_id, query, response, created_at Captures each request-response cycle that the verification pipeline analyzes.
verified_claims interaction_id, claim_text, verdict, confidence, source_citation Stores claim-level verification output for a generated response.
contradictions interaction_id, chunk references, score, created_at Tracks evidence passages that conflict strongly enough to require review.
interaction_citations interaction_id, document reference, chunk reference, source label fields, created_at Associates a response with the exact authorities and evidence chunks it cited.
conversation_state conversation_id, summary, last_updated_at Keeps a resumable summary of the current thread without recomputing the full history.

Relationship rules visible in the ERD

From ERD
  • One user can own many conversations and perform many interactions.
  • One conversation contains many messages and many interactions.
  • One conversation has a single conversation_state record for the current summary snapshot.
  • One interaction can produce many verified_claims.
  • One interaction can surface many contradictions when evidence chunks conflict.
  • One interaction can reference many interaction_citations to preserve source traceability.

This ERD is stronger than the earlier generic model because it shows how verification output is persisted around a real user workflow instead of only listing abstract domain objects.

Ethics and governance

From summary
  • Attorney-client privilege and local processing are first-order design constraints.
  • False negatives are treated as especially harmful because missed hallucinations can create legal risk.
  • Auditability is supported through saved interactions, evidence passages, and verdict explanations.
  • Deployment decisions must balance privacy, quality, cost, and offline capability.

Operational constraints

From summary
  • Local Llama quality may trail larger hosted models on complex reasoning, so verification must compensate.
  • Claim decomposition recall remains imperfect, meaning some claims may still escape review.
  • Hardware needs are non-trivial: 16 GB RAM minimum, 32 GB recommended for production.
  • SQLite write concurrency is limited, though still appropriate for most firm-scale use.
